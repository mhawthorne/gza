### F1 — Reason-code set is out of sync with what the code emits — recommend: **spec gap (reconcile)**

**Status:** ratified contract (OQ#3: reason codes are a stable enumerated set automation may
branch on). So this gap matters.

**Spec** (`lifecycle-engine.md` § Parked reason codes) defines a canonical set. **Code**
emits a different set (`advance_engine.py` lines 807–2661; `recovery_engine.py` lines
985–1014). They diverge three ways:

**(a) Renamed — same concept, different string (8):**

| Spec canonical | Actually emitted |
|---|---|
| `explore-dangling` | `explore-needs-follow-up-decision` |
| `scope-out-of-scope` | `project-scope-violation` |
| `scope-unverifiable` | `project-scope-unverified` |
| `rebase-moot-incomplete-lineage` | `branch-already-rebased-lineage-incomplete` |
| `review-refresh-blocked` | `stale-review-needs-manual-refresh` |
| `review-verdict-unknown` | `review-verdict-needs-manual-attention` |
| `recovery-retry-limit-reached` | `retry-limit-reached` |
| `recovery-manual-review-required` | *(folded into `retry-limit-reached`, see F2)* |

**(b) Emitted but absent from the spec's "closed" set (7) — spec under-enumerated:**
`closing-review-needs-manual-refresh`, `merge-source-needs-manual-resolution`,
`verify-noop-improve-branch-tip-unavailable`, `verify-noop-improve-diff-probe-unavailable`,
`automatic-recovery-disabled`, `manual-failure-reason`,
`newer-recovery-descendant-needs-attention`.

**(c) In the spec but never emitted (2) — spec invented:**
`review-inconsistent-followups` (the zero-follow-ups case actually routes through the
generic `review-verdict-needs-manual-attention` path, `advance_engine.py:2652-2663`) and
`plan-held` (the held-plan `awaiting_human` action was not observed to carry a reason slug).

**Recommendation:** primarily a **spec gap** — the code is the working reality and the spec
was just written. Reconcile the spec's canonical table to the emitted strings (or
deliberately rename in code), add the 7 missing codes, and drop/realign the 2 invented ones.
This is the reconciliation the spec's own implementation-note anticipated — now it's
itemized.

### F2 — `retry-limit-reached` and `manual-review-required` are conflated, leaving dead watch logic — recommend: **code bug (or spec relax)**

**Spec** treats `recovery-retry-limit-reached` and `recovery-manual-review-required` as two
distinct reason codes. **Code:** `recovery_engine.py:989` maps **both** the
`retry_limit_reached` and `manual_review_required` reason codes onto the single emitted slug
`retry-limit-reached`. But `watch.py:99` defines
`_WATCH_PARKED_NEEDS_ATTENTION_REASONS = {"retry-limit-reached", "manual-review-required"}`
and branches on it (`watch.py:163-173, 1238-1244`) to *reuse a parked action instead of
re-spawning a worker*.

**Why it matters:** since the engine never emits `manual-review-required`, watch's check for
that slug is **dead code** — a manual-review-required stop is only handled because it was
already collapsed into `retry-limit-reached`. If anyone later separates the two (as the spec
says they are), the parked-reuse behavior would silently change. The contract (two codes) and
the implementation (one code, plus a watcher that thinks there are two) disagree.

**Refutation attempted:** confirmed the engine has no other emission site for
`manual-review-required`; the watch-set entry has no matching producer. Survives.

**Recommendation:** decide the intent (OQ-adjacent): either emit the two codes distinctly
(code change, matches spec) or collapse them in the spec too and remove the dead watch entry.
