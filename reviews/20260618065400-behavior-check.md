## Findings (divergences)

### WS-S6-PER-CYCLE-HUMAN-REQUIRED-VISIBILITY — DIVERGES — recommend: code bug
**Spec** (`specs/behavior/watch-supervisor.md` S6 / 2A): for every watch cycle, `watch` MUST emit an operator-visible `Needs attention` signal for every in-scope owner whose shared failed-task recovery decision parks it for human intervention, and `--restart-failed` / `--show-skipped` MUST NOT gate that visibility.

**Code** (`src/gza/cli/watch.py:2175-2182`, `src/gza/cli/watch.py:2283-2290`): the recovery loop drops `decision.action == "skip"` rows unless `show_skipped` is enabled, and even then it emits a `SKIP` line rather than `ATTENTION`. Default watch therefore keeps manual-only parked failed recoveries silent on a cycle with no new transition.

**Why it matters:** a human-required failed owner can exist in the shared recovery surface but not appear in the live watch signal surface unless the operator opts into skipped diagnostics.

**Refutation attempted:** checked the other watch attention paths. `_collect_unhandled_failures(...)` only inspects `_iter_status_transitions(...)` (`src/gza/cli/watch.py:1238-1272`), so it cannot provide the standing per-cycle signal for older already-failed rows.

**Recommendation:** code bug — emit shared needs-attention recovery skips through `log.emit_attention(...)` every cycle regardless of `--restart-failed` / `--show-skipped`.

### WS-S6-OWNER-BASED-PARITY — DIVERGES — recommend: code bug
**Spec** (`specs/behavior/watch-supervisor.md` S6 / 2A): parity is owner-based. The compared set is lineage-owner / merge-unit IDs, and the failed leaf ID is detail within the owner's signal rather than a separate top-level attention row.

**Code** (`src/gza/cli/watch.py:2257-2280`, `src/gza/cli/watch.py:208-209`, `src/gza/advance_engine.py:1192-1199`): the emitted recovery attention path renders `_watch_needs_attention_message(failed, ...)`, and the shared recovery attention action is created with `subject_task_id=task.id` where `task` is the failed leaf. The attention key is also leaf-based (`recovery-attention:{failed.id}:...`).

**Code (comparison surface)** (`src/gza/cli/watch.py:1530-1545`, `src/gza/lineage_query.py:1033-1056`, `src/gza/cli/query.py:1656-1659`): watch and `gza incomplete` already start from owner rows, and incomplete re-roots needs-attention display on the owner/declared subject task. The recovery attention emission path in watch does not preserve that owner-based identity.

**Why it matters:** the watch surface can identify the failed leaf as the primary attention row instead of the merge-unit owner that operators act on, breaking the spec's parity contract with `gza incomplete`.

**Refutation attempted:** checked for a later owner-normalization step in the recovery attention emission path; none exists after `_watch_needs_attention_message(failed, ...)`.

**Recommendation:** code bug — emit recovery attention keyed to the owner row and include the failed leaf only as message detail.
